- [severity: blocker] [Header layout] The 128-byte v1 prefix plus `key_mode` (1 byte), one `recipient_id` (16 bytes), and two 96-byte slots totals 337 bytes before reserved padding, so the stated fields cannot fit a 256-byte header -> Publish a byte-accurate layout and either enlarge the header or redesign each self-contained slot to fit.
- [severity: blocker] [Wrap suite] The draft alternates between HPKE and libsodium sealed boxes without freezing the KEM, KDF, AEAD, mode, suite identifier, ciphertext encoding, and absent-slot representation, although these are distinct wire protocols with different context capabilities ([RFC 9180](https://www.rfc-editor.org/rfc/rfc9180.html), [libsodium sealed boxes](https://doc.libsodium.org/public-key_cryptography/sealed_boxes)) -> Select one exact construction, assign an immutable RAO wrap-suite ID, and publish byte-level test vectors before freezing v2.
- [severity: major] [Header authentication] Contrary to the draft, v1 already binds every canonical 128-byte header byte through `SHA-256(header)` in HKDF within `seal.rs`, `open.rs`, and `range.rs`, despite `stream.rs` using empty AEAD AAD, as confirmed by the `key-id-swapped` negative vector -> Specify whether v2 preserves this header-hash key derivation, adds full-header metadata AAD, or replaces it, with domain-separated transcripts covering version, content suite, wrap suite, mode, and every slot.
- [severity: major] [Mode confusion] Because `key_mode` is unauthenticated until a key has been selected and the metadata tag verified, an attacker can force an automatic reader to consult the symmetric registry or unwrap path even though correct full-header binding prevents successful decryption, while current Sutradhara instead selects v1 keys from catalog `Representation` and `key_epoch` -> Make v2 envelope-only or expose separate expected-mode entry points, reject catalog/header disagreement, and never fall back between registry and envelope modes after failure.
- [severity: major] [Seal ordering] The stated sequence “seal with DEK, then wrap DEK, then emit header” is incompatible with using the finalized header containing wrapped slots as metadata AAD or HKDF context -> Generate and wrap the DEK first, serialize the canonical final header, and only then derive content keys and encrypt metadata and payload.
- [severity: major] [Slot binding] Wrapping only the bare DEK permits a valid slot to be transplanted between objects for the same recipient, causing `rao-unwrap` to emit a wrong but successfully unwrapped key before the content metadata tag later rejects it -> Bind a protocol label, object ID, recipient ID, slot index, version, and wrap suite into HPKE `info`/AAD, or embed and verify that context inside a sealed-box plaintext.
- [severity: major] [Randomness] The current seal path is deliberately deterministic and `remanence-aead` has no OS-RNG dependency, so the draft does not yet establish the fresh random DEK and independent per-slot encapsulation randomness on which its security depends, while deterministic `hkdf_salt` is safe only under that freshness invariant ([RFC 9180 randomness requirements](https://www.rfc-editor.org/rfc/rfc9180.html#section-9.2.3)) -> Use a fallible OS-backed CSPRNG, fail sealing when entropy is unavailable, and enforce a fresh DEK per object plus fresh encapsulation randomness for every recipient slot.
- [severity: major] [Object authenticity] HPKE Base and sealed boxes allow anyone possessing the recipient public key to construct an internally valid replacement RAO object, so the proposed catalogless recovery property provides confidentiality and self-consistency but not provenance or substitution detection -> Authenticate an escrowed manifest or stored-object digest with a separately governed signing mechanism, or explicitly declare trusted provenance outside the self-contained format.
- [severity: major] [Escrow trust] Either recipient private key can silently decrypt every past object for its generation and can unwrap the DEK, add another recipient, and produce a validly reauthenticated object under any AAD design, while full-header binding blocks only actors lacking a DEK -> State this unilateral escrow authority explicitly and use dual control, threshold recovery, or an independently signed immutable recipient-set policy if silent access or recipient addition is unacceptable.
- [severity: major] [Hot-host blast radius] The claim that a fully compromised archive host cannot read v2 copies applies only to previously sealed objects whose transient DEKs escaped capture, because that host still holds all preserved v1 archive roots, hdcache and future hot-backup roots, plaintext-pool access, recipient public keys, and the ability to steal future DEKs or substitute future recipient keys -> Replace the blanket claim with a time-scoped attacker matrix and pin recipient-key fingerprints through an offline-approved manifest with drift alarms.
- [severity: major] [Key material] Existing `RootKey`, `DerivedKeys`, and CLI key buffers use `zeroize`, but the draft leaves generated DEK and ephemeral-secret types unspecified and makes the air-gapped DEK file a persistent plaintext decryption capability whose unlinking does not guarantee erasure -> Keep DEKs and ephemeral secrets in zeroizing non-cloneable containers and have `rao-unwrap` create an exclusive `0600` file on controlled volatile or encrypted storage with an explicit post-restore destruction procedure.
- [severity: minor] [Decryption semantics] A wrong DEK fails the metadata AEAD before plaintext output and each payload chunk is authenticated before that chunk is written, but `open.rs` can emit earlier authenticated chunks before a later tag, footer, fill, or digest failure and therefore is neither whole-object atomic nor constant-time across failure stages -> Require restore callers to stage output and publish it only after complete success, as Sutradhara currently does, and document the guarantee as per-frame fail-closed rather than whole-stream constant-time.
- [severity: minor] [Verification coverage] The current RAO-header proof explicitly excludes exact bytes and header hashing, the framing kernel hardcodes 128-byte offsets, the fuzz target parses only a padded v1 header, and the negative vectors do not exercise wrap or cross-mode behavior -> Extend proofs and tests to cover v1/v2 disjoint parsing, exact AAD/HKDF transcripts, mode/version/suite flips, slot transplantation and addition, recipient-ID mismatches, malformed encapsulations, wrong recipients, and both header lengths.
