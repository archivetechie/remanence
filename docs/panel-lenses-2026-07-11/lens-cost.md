Findings below. I confirmed: RAO-AEAD objects are per-bundle (multi-GB) AND per-asset in hdcache (`AEAD_REPRESENTATION="rao-aead-v1"`, keyed by content_sha256); `aead-framing` hardcodes `def HeaderLen := 128` used via `rw [header_len_val]` in ~15 offset lemmas plus a `checked_sub(144)` (H+F) literal pinned by drift-guard; remanence already owns `ring`+`chacha20poly1305`+`hkdf`+`sha2` but has no hpke/sodium/age crate and no `age` binary installed; pilot+console DBs hold ~6 sealed bundles total; no wrapped-key/recipient columns exist in the catalog today.

- [major] [§OQ1 / Impact-map:Proofs] The "append a fixed block after the v1 128B header" v2 variant does NOT shrink the `aead-framing` re-derivation, because `HeaderLen` feeds every offset formula (`cipher_offset`, `stored_size`, `footer_offset`, `inspect_geometry`) as a concrete literal `128`, so any effective H change re-proves A3/A4/A7/A8 regardless of framing — and a flat 128→256 literal swap will be re-paid again at v3. → Parametrize `HeaderLen` in the `aead-framing` extraction (prove `∀ H`) once, which makes v1/v2 coexistence and all future header growth cost zero further re-derivation; append-after only helps `rao-header` (keeps its 128B-core round-trip valid for both versions), not `aead-framing`.

- [major] [§Threat model:overhead / Impact-map] The overhead-honesty argument weighs only multi-GB bundles and omits that hdcache seals per-asset `rao-aead-v1` objects (high count) and PFR ranged-restore reads header geometry, so binding 256B/`FORMAT_VERSION=2` universally would tax the hottest high-count path with a wasted 96B empty slot and force version-dependent offset math on the cache/PFR read path. → State explicitly that v2/256B is scoped to envelope objects only and that hdcache plus all non-envelope symmetric pools keep emitting v1/H=128 headers, so the doubling and the offset churn never touch the hot per-asset path.

- [major] [§Standalone unwrap tool] `rao-unwrap` is a bespoke Rust bin (ring/chacha/hkdf tree) that must be rebuildable on an air-gapped machine in ~2041, yet no stock recovery tool (`age` binary) is even installed today, making the 15-year DR recovery path hinge on reproducing the remanence toolchain. → Define the belt-and-suspenders escrow EXPORT as a stock-`age` file (static binary, distro-packaged, frozen format) so disaster recovery can proceed with owned/off-the-shelf tooling even if `rao-unwrap` cannot be rebuilt.

- [minor] [§OQ2] HPKE-vs-sealed-box is framed as a new-dependency choice, but `ring` is already owned (via `rustls` `tls-ring` in api/daemon) and `remanence-aead` already pulls `chacha20poly1305`+`hkdf`+`sha2`, so a DHKEM(X25519,HKDF-SHA256)+ChaCha20Poly1305 single-shot seal is buildable with zero new crates. → Choose between zero-new-dep primitives-from-`ring` (max reuse, hand-roll audit burden) and `rust-hpke` (one maintained crate, RFC-9180 test-vector conformance), and explicitly reject libsodium (C dependency) and the `age` crate (heavy tree) as the non-boring options.

- [minor] [§OQ2 / OQ1] Storing an `age`-format ciphertext directly in the header slot cannot meet the fixed 96B budget — a single-recipient age file is ~180–220B (text header + X25519 stanza + payload tag), which would blow the 256B header and add proof churn. → Keep the in-header slot as raw HPKE bytes (96B) and reserve the `age` format for the external escrow export, capturing the owned-tooling durability win without breaking the header size budget or its proofs.

- [minor] [§OQ5] "Leave v1 forever vs opportunistic re-seal" is free either way TODAY (~6 sealed bundles total across pilot+console DBs), so this is purely a production-scale tape-time question, and a standalone re-seal pass over the whole AEAD tier at ~300MB/s would re-read+re-write every object for nothing. → Resolve OQ5 now to "leave v1 forever (escrow the registry key) and re-seal ONLY when a tape-generation migration is already reading+writing the bytes" — never a dedicated re-seal pass.

- [minor] [§Multi-recipient] The second recipient's cost is stated as ~96 header bytes, but the catalog operational-copy and escrow-export schemas are net-new (no wrapped-key/recipient columns exist today) and would need to be collection-shaped with per-slot recipient generations, and retire-generation GC becomes per-slot. → Make the header slot array-shaped AND the catalog/escrow copies collection-shaped at first introduction so adding/rotating recipients never forces a later schema migration.

- [nit] [§Impact-map] The re-derivation checklist omits pinned magic literals that a header-length change silently invalidates: `checked_sub(144)` (H+F) in production `inspect`, its drift-guard string-match, and `rao-header`'s `assert_eq!(header_len,128)` / `RAO_HEADER_LEN=128` pins. → Enumerate these pinned literals (144→272, the two 128 pins, and each `rw [header_len_val]` site) in the impact map so the drift guards are updated deliberately rather than tripping mid-implementation.
