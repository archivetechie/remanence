I have enough grounding in both the design and the real code. Here are my findings.

---

- [blocker: §Multi-recipient] Multi-recipient is labeled "recommended, panel to confirm," so a single-recipient build ships a format where losing one generation's air-gapped private key permanently and silently destroys every object sealed to it (undetectable until a restore is attempted years later) → make two independent recipient slots (offsite + escrow) MANDATORY for any envelope pool, and refuse to seal in envelope mode unless ≥2 distinct-custody slots wrap successfully.

- [blocker: §Seal/restore, §Redundant storage] The restore ceremony delivers a per-object DEK key file to the HOT server (archive_restore.py runs `rem archive extract --key-file` on the online host, not the air-gapped one), yet the doc says nothing about how that DEK crosses back over the air gap, where it lands, or when it is destroyed — reintroducing online decrypt capability with no specified handling → specify DEK transport medium (single-use, wrapped), require it to materialize only in tmpfs 0600 (as `materialized_root_key` already does via `/run`/`/dev/shm`), and mandate shred-after-extract, all in the runbook.

- [major: §Authentication of the wrapped key] The claim "v1 binds hkdf_salt-derived context only" is factually wrong — v1 uses EMPTY AAD (`encrypt_metadata`/`encrypt_chunk` pass `aad: &[]`) and binds the FULL 128-byte header via `header_hash()` fed into `derive_keys(root_key, salt, header_hash)`, so the proposed metadata-frame AAD is either redundant (if v2 keeps the full 256-byte header in `header_hash`) or a downgrade (AAD protects only the metadata frame, not payload) → pin the rule as "`header_hash` covers the entire 256-byte v2 header including `wrapped_dek`, computed after wrapping," and drop or justify the separate AAD change rather than build a weaker binding.

- [major: §Header, §Impact map] The keyless verify/footer geometry in `crates/remanence-aead/src/inspect.rs` (and the A7 aead-framing kernel) hardcodes `RAO_HEADER_LEN = 128`, so a verify-sweep or `inspect` over a v2 (256-byte header) object computes the wrong footer offset and mis-detects completion on mixed tapes → make the geometry version-keyed on the parsed header length and add a mixed-tape (v1+v2) verify-sweep negative vector before writing any v2 object.

- [major: §Header (v1 readers reject on version)] Old rem binaries hard-reject v2 (`InvalidHeaderLength` + `UnsupportedFormatVersion`), so if a multi-object-tape reader aborts the whole restore on first parse error, one v2 object strands every v1 object on that tape for un-upgraded readers → state the rollout-ordering rule explicitly ("upgrade ALL readers/verify hosts to v2-capable before any v2 object lands on a shared or multi-object tape") and make the multi-object reader skip-and-continue past unparseable objects rather than abort.

- [major: §Seal / restore flows] The seal flow discards the plaintext DEK on completion but never states the crash-window invariant, so a crash after the wrapped key is written to a footerless (incomplete) object leaves the source needed for re-seal unprotected and can leave an orphan catalog/escrow row pointing at a wrapped key for an object that will be re-sealed under a DIFFERENT random DEK → require source retention until the footer is confirmed AND the wrapped key is durably recorded, and specify reconciliation that GCs wrapped-key rows for footerless/incomplete objects.

- [major: §Standalone unwrap tool, §Open Q4] Nothing detects which recipient generations are still referenced by live objects, so an operator cannot safely decide when a generation's private key may leave the safe, and objects wrapped to a retired generation go unnoticed → commit open-Q4 (verify-sweep parses v2 headers and reports recipient-generation coverage) as a required control, and gate any private-key destruction on "zero live objects reference this generation."

- [major: §Multi-recipient] Partial-slot-wrap behavior is unspecified (slot 1 wraps, slot 2's configured pubkey is bad) — a silent degrade to single-recipient invisibly removes the dual-recovery guarantee for exactly the objects that hit the bad config → fail the seal closed on any configured slot failing, and record in the catalog the recipient generations each object was ACTUALLY wrapped to (not the intended config) so dual-coverage is queryable.

- [major: §Standalone unwrap tool, §the drill] `rao-unwrap` is shipped as a static bin and the quarterly drill "exercises it end to end," but a drill that only re-runs the cached binary proves it still runs, not that it can be REBUILT on a future OS when that binary finally won't (15-year tool bit-rot) → archive pinned source + toolchain alongside the escrow and make the drill periodically REBUILD `rao-unwrap` from that archived source, not just execute the cached artifact.

- [major: §the drill] The drill is specified as a one-line pointer with no acceptance criteria, so it cannot be shown to catch safe-access rot, media rot, or people rot → specify it as: a first-timer restores a real object sealed >1 quarter earlier from a COLD medium using only the safe's private key and the written runbook, and confirms every referenced generation's private key is physically present and readable in the safe.

- [major: §Escrow export lifecycle] Objects sealed AFTER the last escrow export have their redundant wrapped-key copy only in the (corruptible) header + catalog until the next export, and nothing alarms on this staleness → specify export cadence/trigger and a "objects sealed since last successful escrow export" counter that alarms past a threshold.

- [major: §Open Q5 / migration posture] "Leave existing v1 AEAD objects as v1 forever (registry key escrowed)" escrows a key derived from the hard-coded dev seed (registry.py `_TEST_SEED`), which is public — so those objects are effectively unencrypted and re-sealing (not escrow) is the only real fix → flag that option (a) is unsafe until the dev-seed is replaced AND surface re-seal-on-migration as the only path that actually makes already-written v1 AEAD objects confidential.

- [minor: §Header] In envelope mode the DEK replaces the registry root key, but `derive_salt` also consumes `root_key` and the KDF labels (`rao1-salt-v1`, `rao1-payload-v1`) are shared with v1-registry derivations with no `key_mode` domain separation → specify envelope-mode salt derivation (DEK as the derive_salt/derive_keys root on both seal and restore, which is what keeps the "restore adapter interface unchanged" claim true) and bump/parameterize the KDF labels so v1 and v2 derivations are provably distinct.

- [minor: §Header] Two wrapped slots share a single `recipient_id[16]`, so the escrow slot carries no generation tag and `rao-unwrap` cannot validate that a presented escrow key is the right generation for slot 2 → give each slot its own `recipient_id`.

- [minor: §Standalone unwrap tool] The doc does not require `rao-unwrap` to validate the presented key against the header's `recipient_id` and emit an actionable mismatch ("this object needs generation N; you supplied generation M"), so a wrong-generation key on disaster day yields an opaque crypto failure → mandate a specific generation-mismatch error message keyed on `recipient_id`.

- [minor: §Header, §Threat model] The v2 footer value and its new completion-detection offset (footer moves +128 bytes) are unstated — the v1 footer literal is `RAO1_STREAM_END.` (contains "RAO1") → state whether v2 keeps that footer literal and confirm completion is detected at the version-derived offset, so scanners and verify tooling agree on where "done" is.

- [nit: §Open Q1] A fixed 256-byte header with two 96-byte slots + recipient tag + reserved leaves no room for a third recipient (e.g., rotating in a new escrow while retaining the old), forcing a v3 bump for a routine key-lifecycle operation over a 15-year format → weigh a length-prefixed key frame (TLV) for the wrapped-key area even if the scalar header stays fixed.
