I have both documents. The design doc lives on the `rao-wrapped-key-header` branch (not checked out); I read it via `git show`. Here is my walkthrough-lens review.

- [blocker] [§Standalone unwrap tool] "rao-unwrap ... ships with the DR runbook" escrows a document, not a runnable — a Rust bin on libsodium/HPKE won't build or launch on a 2036 OS, so a survivor holding tapes+keys+passphrase still can't unwrap → FIX: escrow a static, version-pinned rao-unwrap binary AND its reproducible source/toolchain image, and prove it runs cold at every drill.
- [blocker] [§Seal / restore flows] The final "DEK → files on disk" step "hands the DEK to the existing restore adapter," which the companion consult confirms is AssetLocator/catalog-coupled, so the design's "recoverable ... with no catalog" claim collapses at the last step → FIX: ship a standalone header→DEK→plaintext-bytes decrypt tool (sibling to rao-unwrap) needing no catalog/sutradhara, and make it the drilled path.
- [blocker] [§escrow ceremony (referenced, unwritten)] The entire human recovery ceremony is deferred to an unwritten "DR runbook"/"escrow thread"; neither doc gives the numbered steps a stressed non-designer would follow → FIX: write and version the runbook as an explicit ceremony (locate → fetch key → shuttle header → unwrap → decrypt → verify), stored in the safe and the escrow kit.
- [blocker] [§escrow ceremony] A 2-of-2 owner+one-other passphrase split with no succession/re-split is LESS survivable than a single holder — any one departure, death, or forgotten share = permanent silent loss → FIX: use Shamir M-of-N (e.g. 2-of-3) with named custodians and a documented succession + re-split procedure.
- [blocker] [§Multi-recipient / paper] What is ON the paper is never specified; a raw private key on one sheet means a single photograph reads the whole archive, while the passphrase filed beside the sealed export makes the safe a single point of compromise → FIX: specify paper contents exactly (recipient inventory + generation labels + locate steps), with key material only ever as split shares in separate custody, never a whole key on one sheet.
- [major] [§Decision record #2 / Threat model] "Self-describing ... recoverable ... with no catalog" answers what an object IS, not WHERE it is; with the host gone the owner cannot tell which of N tapes holds the priority footage → FIX: keep a catalog/index snapshot on the LAN backup box and in escrow plus a per-tape object manifest in the runbook (an index does not weaken the air gap).
- [major] [§Header / recipient_id] recipient_id[16] is opaque bytes with no registry mapping it to a human ("generation 1, blue envelope, safe A"), so the wrong-generation fumble ends in a hex mismatch the stressed owner cannot act on → FIX: maintain an escrowed generation registry (recipient_id hex → label, create/retire date, physical location) and have rao-unwrap print the needed label.
- [major] [§Header] key_id "carries the recipient generation" AND recipient_id "names the ... generation" — two 16-byte fields both claiming generation, with no statement of which rao-unwrap authenticates against → FIX: designate one authoritative generation field and specify exactly what rao-unwrap checks and reports.
- [major] [§Open questions #4 / Multi-recipient] Nothing gates recipient-generation retirement on "no live object still wrapped to it," so routine key hygiene can destroy the only private key that decrypts old objects → silent unrecoverability → FIX: make retirement a hard gate (block retire / force re-wrap while any object references the generation), mirroring durability-floor enforcement.
- [major] [§Open questions #5] There is no re-wrap-on-rotation path, so every generation's private key must be escrowed forever (unbounded escrow, growing theft surface) or old objects strand → FIX: define a re-wrap ceremony (old key unwraps DEK → re-seal to new recipient) and a retention policy for retired private keys.
- [major] [§Standalone unwrap tool] "the drill (quarterly) ... exercises it end to end" is one clause with no roles, script, duration, or definition of "pass" → FIX: specify the drill as a signed checklist (who, steps, expected artifact hashes, pass = plaintext byte-match of a canary object) with recorded evidence.
- [major] [§Standalone unwrap tool] The drill detects but the design creates no repair loop — safe-combo-forgotten / passphrase-holder-gone / tool-won't-run have no remediation or owner → FIX: wire drill failure to a STANDING-style escalation with named remediation playbooks (rebuild+re-escrow tool, re-split passphrase, re-cut paper) and a due date.
- [major] [§escrow ceremony] Silent unrecoverability is caught only if the drill RECONSTRUCTS the passphrase from the actual shares each time; a drill using a convenience copy never detects a lost/corrupt share → FIX: mandate that every drill reassembles the passphrase from custodians' shares and opens the sealed export from scratch.
- [major] [§Standalone unwrap tool] "must run on an air-gapped machine" but nothing preserves that machine's environment — proving rao-unwrap runs on the dev box proves nothing about a bare 2036 recovery box → FIX: drill on a cold/representative offline machine (or a preserved bootable image) and treat "won't run on a clean OS" as a drill failure.
- [minor] [§Seal / restore flows] The unwrapped DEK returns to the hot restore machine in plaintext and the ceremony never says to destroy it → FIX: add explicit DEK-shred and scope ("single restore, then wipe") steps to the runbook.
- [minor] [§Header] The 256-byte header is all binary; a survivor doing a hexdump can read nothing human → FIX: reserve a short ASCII field (generation label + seal date) in the non-secret header extension for eyeball diagnosis.
- [minor] [§Standalone unwrap tool] The design never says rao-unwrap tries both recipient slots or reports which matched, so a two-slot object with the wrong key gives an unhelpful failure → FIX: have rao-unwrap enumerate both slots, auto-select the match, and on failure print "object wants <label A>/<label B>, you supplied <label C>."
- [nit] [§Standalone unwrap tool] A quarterly FULL ceremony (two custodians + safe + air-gapped machine + tape read) risks decay and skips over 10 years → FIX: split into a heavy annual end-to-end drill plus light quarterly checks (tool boots, shares present, safe opens, canary header parses).
- [nit] [§Threat model] The walkthrough silently assumes an LTO drive that can still READ the archived generation exists in 2036 (LTO back-read is ~2 generations) → FIX: note tape-drive/media longevity as an escrow-kit concern (retain a known-good drive or set a migration cadence).
