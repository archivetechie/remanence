Key confirmations: the design doc does not exist at the stated path (searched exhaustively); the real header is 128 bytes with a fixed 64-byte object_id field; `RAO_HEADER_LEN` is baked into offset/size math and `header_hash` feeds `derive_keys`; the entire seal/CLI/registry stack is symmetric 32-byte-root only; the frozen negative-vector corpus pins format_version=2 as rejected; and no `rao-unwrap`, escrow export, version probe, or recipient-keypair concept exists anywhere.

---

- [blocker: §whole-design] The reviewed artifact `remanence/docs/design-rao-wrapped-key-header.md` does not exist anywhere on disk (only the sutradhara consult doc does), so there is no single source of truth to field-check and every downstream prompt would re-invent the contract → write and land the design doc with an explicit wire-layout table and consuming-surface list before any implementation prompt is cut.
- [blocker: §sealer-interface] "Pass recipient pubkey instead of materializing a root key" has zero implementation surface: `remanence-aead::seal()` takes `root_key: &RootKey` (32 bytes only), the CLI is `--key-file <32B> --key-id` (`rem_archive_cli.py:129`), and `KeyRegistry._read_root_key` hard-rejects anything but 32 bytes — this is a new public-key crypto core + CLI verb + registry keypair type, not a header change → scope it as a distinct envelope-sealing subsystem with its own spec, or drop public-key wrapping in favor of a separate symmetric `backup` epoch.
- [blocker: §kdf-binding] `derive_keys(root_key, hkdf_salt, header.header_hash())` binds the SHA-256 of the *entire* serialized header into the payload/metadata keys, so any wrapped-key bytes carried in the header become key-derivation input — re-wrapping the same root to a new recipient generation changes `header_hash` and permanently breaks decryption of already-sealed bytes → the design must exclude the wrapped-key region from both `header_hash` and `derive_salt`, or explicitly forbid re-wrap/re-escrow without full re-seal.
- [major: §versioning/single-source] A 256-byte v2 header simultaneously violates three frozen single-sourced artifacts: negative vector `format-version-2`→`UnsupportedFormatVersion` and `header-len-not-128`→`InvalidHeaderLength` (`fixtures/rao/negative-envelope.json`), Lean `rao-header` H3 (frozen "header length = 128, format version = 1"), and Lean `aead-framing` A3/A4 (constant `H=128` inside `cipher_offset`/`stored_size_from_parts`) → the design must parameterize header length by version, re-cut the conformance corpus + independent verifier, and re-establish both proofs as an explicit deliverable.
- [major: §capability-signaling] No version/capability probe exists in `rem_archive_cli.py` — sutradhara cannot tell whether the installed rem binary supports v2/envelope sealing, so a new `--recipient-pubkey` against an old binary fails as a generic clap arg error, not a clean unsupported-capability signal → add a single-sourced rem capability/version contract (e.g. `rem archive capabilities`) that sutradhara checks before selecting the v2 sealer.
- [major: §key-domain-model] Domains are not symmetric or config-only as the "backup domain unaffected" framing implies: `KEY_DOMAINS` is a closed 2-set, `key_domain()` silently classifies any non-`hdcache-` id as `archive`, and `archive_fanout.py:400` hard-asserts `KEY_DOMAIN_ARCHIVE` — and none of these can represent an asymmetric "recipient generation" keypair (registry stores 32-byte symmetric roots only) → the design must specify where the recipient keypair lives (new store, not `KeyEpoch`), update `key_domain()`'s default-to-archive fork, and add `backup-` prefix validation.
- [major: §restore-adapter] `archive_restore.py:958` restores strictly via `--key-file <32B symmetric root>`; envelope restore needs the recipient *private* key, which the registry does not hold and which the consult doc flags as forcing a hot decrypt secret → the design must define the restore key input (new `--private-key` interface + where the private key is sourced) and resolve the unattended-vs-operator decrypt-secret question before restore is claimed to "take a key file."
- [major: §new-artifacts] The `rao-unwrap` tool and the escrow export format have no existing code or spec anywhere (grep-empty) — their inputs/outputs, encoding, and object linkage are entirely unspecified, and there is no existing wrapped-key export for them to ride → both must be specified as first-class artifacts (byte formats, CLI contracts, error taxonomy) rather than referenced as if they exist.
- [minor: §rao-unwrap] The "wrapped key is in the first 256 bytes" contract holds for object_id (it is a fixed 64-byte NUL-padded wire field at `0x40..0x80`, not variable-length, so a fixed wrapped-key offset is feasible) but is wrong for v1 objects, whose header is 128 bytes with no wrapped key → the tool must dispatch on `header_len`/`format_version` and define behavior for v1 objects rather than unconditionally reading 256 bytes.
- [minor: §version-reject-claim] "v1 readers reject on version" is imprecise: `RaoHeader::parse` checks `header_len == 128` (→`InvalidHeaderLength`) *before* `format_version == 1` (→`UnsupportedFormatVersion`), so a 256-byte v2 object is rejected on length, not version → correct the design's rejection-reason claim and pick the intended axis (a v2 that keeps `header_len=128` is the only one that trips the version check).
- [minor: §suite-signaling] Envelope mode changes the key-derivation origin (unwrap→root vs direct root), which is arguably a new `suite_id` rather than only a `format_version` bump; both `unknown-suite-id` and `flags-nonzero` are frozen negative vectors → decide and document the signaling axis (version vs suite vs a flags bit) and update the frozen corpus accordingly.
- [minor: §byte-budget] "256 bytes" only fits a compact X25519 envelope (~80 B: 32 ephemeral + 48 wrapped) within the 128-byte extension region and does not fit RSA/large-KEM wraps (256+ B) → the design must pin the exact wrap primitive and show the field-by-field byte budget for the 0x80..0x100 region.
- [minor: §catalog] `Copy` has no recipient/wrapped/envelope columns; today the only extensibility point is `storage_metadata` JSON (`cloud_blob.py:131`), while the consult doc directs a dedicated `IntakeBackup` table → the design must state which table owns wrapped-key + recipient-generation, and must not park security-critical key material in the free-form `Copy.storage_metadata` blob.
- [nit: §naming] "recipient generation" collides with two existing unrelated `generation` concepts — `KeyEpoch.generation` (int epoch counter, `registry.py`) and `media_generation` (str, `models.py:777`) → rename to something unambiguous (e.g. `recipient_epoch`/`escrow_generation`) to avoid cross-surface confusion.
- [nit: §framing] "wrapped-key header" plus "first 256 bytes" leaves it ambiguous whether the wrapped key rides inside the RAO header or in a sidecar/prefix artifact; if a sidecar, `header.rs` is untouched and the name misleads → state explicitly whether the wrapped key is in-header (triggering all the versioning/KDF findings above) or a separate artifact.
