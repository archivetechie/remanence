# Formal verification status

This document is the reviewer-facing inventory for Remanence formal proof work.
The detailed theorem statements live in each `verif/*/SPEC.md`; this page says
what is proved, what is only tied by drift guards, and what remains normal Rust
test coverage.

Status reflects the proof-bearing state after commit `4cef8f2`
(`verif: strengthen CRC and AEAD framing proofs`).

## Trust model

The trust anchor for local proof claims is the Lean type checker accepting the
project with no local `sorry`, `axiom`, or `admit` in the target proof files.
Leanstral, Claude, and Codex are proof-search or editing aids only.

Each proof area has two layers:

- A proof-facing Rust extraction in `verif/<area>/src/lib.rs`.
- A Lean proof over Aeneas-generated definitions plus hand-written
  specification theorems in `verif/<area>/lean`.

Most extraction crates also have a Rust `drift_guard` test. That test pins
selected production snippets and selected extraction snippets. It is a useful
change detector, but it is not a semantic proof that the extraction and
production code are identical.

Current Lean builds replay external Aeneas files that warn about `sorry` in
`Aeneas.Std.Slice` and `Aeneas.Std.StringIter`. The local Remanence proof files
listed below are expected to have no local placeholders.

## Proved areas

| Area | Target | Proved claim | Boundary |
| --- | --- | --- | --- |
| `verif/parity-state` | `remanence-parity::model::{ObjectParityState, ObjectParityStateUpdateRange}` | Object parity state classification, error completeness, watermark-advance skip safety, monotonicity, and recompute consistency. | Does not prove Layer-5 transaction scheduling, catalog persistence, or physical parity reconstruction. |
| `verif/parity-mapping` | `crates/remanence-parity/src/mapping.rs` arithmetic extraction | Epoch size, in-bounds coordinates, row-major shape, data-coordinate round trip, and rejection of invalid coordinates. | Proves pure mapping arithmetic, not sidecar encoding or tape IO. |
| `verif/parity-capacity` | `crates/remanence-parity/src/capacity.rs` arithmetic extraction | Sidecar/bootstrap size formulas, epoch completion math, tape reserve, spool reserve, and capacity gate ordering. | Proves arithmetic and branch order under stated no-overflow/invariant hypotheses, not live spool filesystem behavior. |
| `verif/crc64-xz` | `crates/remanence-crc/src/lib.rs` kernel extraction | CRC-64/XZ constants, reflected bit step, table entry, and generated byte update match the bit-vector specification for every relevant scalar input; selected normative vectors are checked. | The arbitrary-length public `crc64_xz(bytes)` slice loop is specified and tested, but the full slice iteration over all byte lists is not yet an all-input generated-code theorem. Sidecar/audit call-site byte ranges are not proved here. |
| `verif/aead-framing` | Pure arithmetic from `crates/remanence-aead/src/{stream,range,inspect}.rs` | Chunk-count validation, payload-frame length, stored-size rounding, ciphertext offsets, plaintext range validation, non-empty range planning, keyless inspect geometry, wrapper equivalence for `expected_stored_size`, and selected fail-closed edges. | Does not prove ChaCha20-Poly1305 security, HKDF, SHA-256, CBOR canonicalization, byte IO, allocation behavior, or full parser/open workflow. |
| `verif/tape-init` | `crates/remanence-api/src/tape_init.rs::decide_tape_init` branch core | Committed-pool conflict dominance, fail-closed BOT decisions, blank BOT rules, clean bootstrap no-op, and ordered Remanence bootstrap hazards. | UUIDs, pool strings, and error payloads are compact proof-facing categories. The proof excludes BOT reading, catalog projection, bootstrap writes, and hardware orchestration. |
| `verif/pool-selection` | `crates/remanence-api/src/pool_selection.rs` ranking kernel | Fit predicate, completion predicate, leftover arithmetic, and pairwise ranking/tie-break order for `CompleteOrFill` and `FillOldest`. | Proves the stable ranking kernel, not Rust iterator internals, `Vec`, trait-object dispatch, catalog projection, drive occupancy projection, or caller footprint projection. |

## Normal-test-only areas

These are intentionally outside the current formal proof surface:

- SCSI/changer/drive behavior and all physical tape interactions.
- Layer-5 daemon session orchestration, cancellation, concurrency, and gRPC.
- SQLite catalog persistence and rebuild paths.
- RAO archive encode/decode round trips.
- Sidecar/footer binary layout beyond the pure arithmetic already listed.
- AEAD cryptographic security and keyed authentication.
- End-to-end write/read/verify scenarios in `~/system`.

Those areas remain covered by Rust unit tests, integration tests, system
scenarios, code review, and operational runbooks.

## How to check the proof inventory

For a full local proof refresh, run each proof crate's drift guard and each Lean
build:

```text
cd verif/<area> && cargo test
cd verif/<area>/lean && PATH=$HOME/.elan/bin:$PATH lake build
```

The current proof areas are:

```text
aead-framing
crc64-xz
parity-capacity
parity-mapping
parity-state
pool-selection
tape-init
```

Repo-level gates still matter even for proof-only/doc-only edits:

```text
cargo test
cargo fmt --check
cargo clippy -- -D warnings
cargo build --release
```

## Practical wording

Use this wording in comments and design docs:

```text
This pure arithmetic/decision kernel has a Lean proof in verif/<area>; see
docs/formal-verification-status.md and verif/<area>/SPEC.md for scope.
```

Avoid this wording:

```text
This code is formally proved correct.
```

The second sentence overclaims. The current proofs are useful and valuable, but
they prove specific kernels under specific extraction and scope boundaries.

## Recommended next target

The next formal proof should start RAO format correctness incrementally:

```text
decode_header(encode_header(x)) = x
```

for the RAO header or manifest core under valid input bounds. That moves toward
the strategic `decode(encode(x)) = x` archive-format claim without trying to
prove the entire archive stack in one step.
