# Formal verification status

This document is the reviewer-facing inventory for Remanence formal proof work.
The detailed theorem statements live in each `verif/*/SPEC.md`; this page says
what is proved, what is only tied by drift guards, and what remains normal Rust
test coverage.

Status reflects the proof-bearing state through the CRC public slice-loop proof,
the fixed-capacity RAO manifest array/fold proof, the RAO manifest planner-fold
and membership-bridge proofs, and the parity sidecar layout and range-bound
bridge proof.

## Trust model

The trust anchor for local proof claims is the Lean type checker accepting the
project with no local `sorry`, `axiom`, or `admit` in the target proof files.
Leanstral, Claude, and Codex are proof-search or editing aids only.

Each proof area has two layers:

- A proof-facing Rust extraction in `verif/<area>/src/lib.rs`.
- A Lean proof over Aeneas-generated definitions plus hand-written
  specification theorems in `verif/<area>/lean`.

Most extraction crates also have a Rust `drift_guard` test. That test pins
selected production snippets and selected extraction snippets. It is a useful
change detector, but it is not a semantic proof that the extraction and
production code are identical.

Current Lean builds replay external Aeneas files that warn about `sorry` in
`Aeneas.Std.Slice` and `Aeneas.Std.StringIter`. The local Remanence proof files
listed below are expected to have no local placeholders.

## Retired proof areas

The registry-symmetric `verif/aead-framing`, `verif/rao-header`, and
`verif/rao-archive` proof crates were retired on 2026-07-17 when that encrypted
representation was removed. Their drift-guard-pinned production snippets changed
under the envelope geometry, so the old theorems no longer described the shipped
code. The proof crates and guards were deleted; no Lean statement was weakened
or relabeled as current envelope coverage.

The replacements are tracked as **RAO-FORMAL-PREFIX** and
**RAO-FORMAL-HEADER-KEY-FRAME**.

## Normal-test-only areas

These are intentionally outside the current formal proof surface:

- SCSI/changer/drive behavior and all physical tape interactions.
- Layer-5 daemon session orchestration, cancellation, concurrency, and gRPC.
- SQLite catalog persistence and rebuild paths.
- RAO header/key-frame/prefix geometry and archive composition beyond the
  retained metadata and manifest proof surfaces.
- Production sidecar byte-buffer construction/parsing beyond the proved fixed
  scalar layout, CRC-window, and range-bound arithmetic.
- AEAD cryptographic security and keyed authentication.
- RAO key-frame geometry and header/key-frame codec proofs, tracked as
  **RAO-FORMAL-PREFIX** and **RAO-FORMAL-HEADER-KEY-FRAME**.
- End-to-end write/read/verify scenarios in `~/system`.

Those areas remain covered by Rust unit tests, integration tests, system
scenarios, code review, and operational runbooks.

## How to check the proof inventory

For a full local proof refresh, run the inventory gate:

```text
make proof-inventory
```

This discovers each proof crate and runs:

```text
cargo test drift_guard -- --list
cargo test drift_guard
cargo test
lake build
maintained Lean placeholder scan
```

The current proof areas are:

```text
crc64-xz
parity-capacity
parity-sidecar-layout
parity-mapping
parity-state
pool-selection
rao-manifest
rao-metadata

```

Repo-level gates still matter even for proof-only/doc-only edits:

```text
cargo test
cargo fmt --check
cargo clippy -- -D warnings
cargo build --release
```

## Practical wording

Use this wording in comments and design docs:

```text
This pure arithmetic/decision kernel has a Lean proof in verif/<area>; see
docs/formal-verification-status.md and verif/<area>/SPEC.md for scope.
```

Avoid this wording:

```text
This code is formally proved correct.
```

The second sentence overclaims. The current proofs are useful and valuable, but
they prove specific kernels under specific extraction and scope boundaries.

## Recommended next target

The next formal work is **RAO-FORMAL-PREFIX** followed by
**RAO-FORMAL-HEADER-KEY-FRAME**. These must start from fresh envelope
extractions and cover key-frame-dependent prefix/range geometry plus the exact
scalar-header/key-frame codec boundary. The retained metadata and manifest
proofs remain useful, but there is currently no live formal claim for the
header, prefix geometry, or archive-level composition.

Note (2026-07-18 audit): the writer-schema *numbers* in the rao-manifest
and rao-metadata proofs (map lengths, version values, digest byte length,
entry counts) are proved equal to the extracted module constants, not to
independent literals; if a constant were extracted wrongly, the round trips
would move with it. Those numbers are anchored by the drift-guard tests
pinning the production source, which is the intended division of labor.
Tape-init's coverage is the enumerated decision tuples (point-checks), not a
quantified decision-table invariant.
