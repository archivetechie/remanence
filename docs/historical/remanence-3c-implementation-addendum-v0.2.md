# Remanence Layer 3c / rem-tar Implementation Addendum

**Status:** Normative implementation addendum, v0.2  
**Applies to:** `layer3c-design.md` v0.4.4 and `rem-tar-v1-design.md` v0.9.3  
**Supersedes:** `remanence-3c-implementation-addendum-v0.1.md`  
**Purpose:** Close the remaining pre-go-live issues raised in external review: sidecar-header fragility, catalog-less map reconstruction, sidecar clustering, crash/restart rebuild bounds, bulk recovery performance, drive-compression preconditions, decoder ownership, object size limits, and hardware proof gates.

This addendum is part of the implementation contract. It does not change the high-level architecture: object tape files remain clean pax tar archives separated by filemarks; parity remains in sidecar tape files; parity epochs span object boundaries; the catalog remains authoritative for normal operation. It tightens the sidecar wire format, commit semantics, bootstrap/map structures, and recovery APIs so that catalog-less recovery and contiguous-damage recovery are practical.

---

## 0. Decisions from the latest review

The latest review agreed with most of v0.1, especially:

- primary + tail sidecar metadata copies plus a footer locator;
- excluding metadata-copy health from the canonical filemark-map digest;
- adding a sidecar directory to the bootstrap/root-of-trust path;
- adding bulk/epoch recovery instead of relying only on `recover_block_at`;
- disabling and verifying hardware compression;
- owning the GF(2^8) codec rather than defining the tape format by crate internals;
- keeping parity shard payloads unreplicated and relying on the three-copy archive policy for sidecar-payload loss.

This v0.2 incorporates the review's two real objections and a few smaller notes:

1. **No volatile sidecar deferral in v1.** v0.1 allowed up to 64 completed sidecars to remain deferred. That can push restart rebuild from roughly one open epoch to dozens of epochs. v0.2 removes volatile deferral from the v1 default. Completed sidecars generated by an object are emitted at that object's close, before the object commit bundle is catalog-committed.
2. **Add `parity_map` now.** v0.1 assumed the sidecar directory would fit in one bootstrap block for the default scheme and deferred overflow handling. v0.2 defines a `parity_map` tape file kind now, so conservative or custom schemes get the same catalog-less robustness.
3. **Bound bulk-recovery memory.** Bulk recovery is windowed by an operator memory cap; it may take multiple physical passes if the cap is below a full epoch's peer set.
4. **Document small final sidecar proximity.** For tiny final partial epochs, primary/tail/footer metadata copies can be physically close. This is accepted; the bootstrap/parity-map directory still preserves map structure.
5. **Make compression verification drive-specific.** The implementation must prove the actual MODE SENSE / MODE SELECT path on the deployed LTO-9 drive models.

The main pushback: v0.2 does **not** lower `max_deferred_sidecars` to 8-16. It removes volatile deferral entirely for v1. Deferral is a future optimization only if the parity spool is made crash-durable or if the implementation explicitly accepts and budgets a multi-epoch resume rebuild.

## 1. Summary of required changes

Implement these changes before live-tape go-live:

1. **Replicate sidecar header/index metadata.** Every parity sidecar must carry a primary header/index set at the beginning and a tail header/index copy at the end, followed by a one-block footer locator.
2. **Add sidecar-directory root-of-trust.** Bootstraps must either carry an inline `SidecarEpochDirectory` or point to a validated `parity_map` tape file containing that directory.
3. **Define `parity_map` tape files now.** Any directory that does not fit comfortably in the bootstrap block must be written as a separate filemark-delimited control tape file. Conservative/custom schemes must not silently omit sidecar directory robustness.
4. **Change catalog-less scan reconstruction.** Scan classification must use primary sidecar header, footer + tail header, bootstrap inline directory, and `parity_map` overlay before validating the canonical map digest.
5. **Do not use volatile sidecar deferral in v1.** Completed sidecars produced by an object are emitted at that object's close. The object and its completed sidecars are committed as one catalog transaction.
6. **Bound restart rebuild.** For v1 committed state, restart rebuild must cover at most the live partial epoch, not an arbitrary backlog of completed but deferred sidecars.
7. **Add bulk/epoch recovery.** Keep `recover_block_at` for one-off errors, but add region/epoch recovery that reads surviving shards once in physical order and reconstructs all affected stripes together.
8. **Make drive compression a hard false precondition.** Layer 3a must configure and verify LTO-DC/hardware compression disabled for all 3c parity-protected writes.
9. **Own the GF(2^8) codec.** Implement the Appendix A RS math directly. Treat `reed-solomon-erasure` only as an optional acceleration backend gated by byte-identical conformance tests.
10. **Enforce one-object-one-tape capacity.** rem-tar objects do not span tapes. Objects too large for an empty tape after parity and reserve are rejected before writing.
11. **Clarify parity overhead naming.** Report both `m/k` and `m/(k+m)` so operators and capacity planners do not mix the two.
12. **Gate live tape on hardware proof.** Power-loss, synchronous filemark durability, open-epoch rebuild, sidecar-header damage, parity-map recovery, drive compression disablement, and bulk recovery must pass on mock tape, QuadStor, and then scratch LTO media.

## 2. Sidecar metadata replication

### 2.1 Problem

The current sidecar model has a bad catalog-less failure mode: if the sidecar's block 0 is damaged, the scanner may demote the sidecar to an object candidate, causing the reconstructed filemark map to disagree with the bootstrap's whole-map digest. One damaged sidecar header can therefore disable catalog-less parity recovery for the whole validated map, even though only one epoch's sidecar was physically damaged.

### 2.2 Required layout

Each sidecar tape file must have this layout:

```text
sidecar tape file:
  block 0 .. H-1                 primary sidecar header/index set
  block H .. H+P-1               raw parity shard blocks
  block H+P .. H+P+H-1           tail sidecar header/index copy
  block H+P+H                    sidecar footer locator
  filemark
```

Where:

```text
H = sidecar_header_block_count
P = parity_shard_block_count = S * m for a full epoch
```

For a final partial epoch, `P` is the number of parity shard blocks actually written for the sidecar. Logical zero-fill data shards are not written to tape.

### 2.3 Header/index set contents

The sidecar header/index set is the complete metadata needed to use the sidecar:

```text
SidecarHeaderIndexSet:
  sidecar magic
  sidecar format version
  tape_uuid
  epoch_id
  scheme_id and scheme parameters
  block_size
  protected_ordinal_start
  protected_ordinal_end_exclusive
  real_data_shard_count
  sidecar_header_block_count H
  parity_shard_block_count P
  sidecar_total_block_count
  primary_header_start_block = 0
  tail_header_start_block = H + P
  footer_block_index = H + P + H
  copy_kind = primary | tail
  copy_generation = 0
  canonical_metadata_hash
  parity index entries
  data CRC entries
  per-block CRC-64/XZ for every header/index block
```

The **canonical metadata hash** is SHA-256 over a deterministic encoding of the sidecar metadata and index entries, excluding copy-local fields and CRC fields. The primary and tail copies must have the same canonical metadata hash.

A copy is usable only if:

```text
sidecar magic matches
header/index CRCs pass
canonical metadata hash matches the footer metadata hash
scheme parameters match the bootstrap/catalog scheme
protected ordinal range is sane and non-empty
```

If both copies are readable but disagree, the sidecar is suspect and must not be used for parity recovery unless an operator explicitly enters a forensic mode.

### 2.4 Footer locator

The final block of the sidecar must be a fixed-size footer locator. It lets a scanner identify a sidecar even when block 0 is damaged.

```text
SidecarFooter:
  magic = HMAC-SHA256(tape_uuid, b"REM\x00PARFOOT\x01")[0..8]
  sidecar_footer_version: u16
  tape_uuid: [u8; 16]
  epoch_id: u64
  protected_ordinal_start: u64
  protected_ordinal_end_exclusive: u64
  sidecar_header_block_count: u32
  parity_shard_block_count: u32
  sidecar_total_block_count: u64
  primary_header_start_block: u64        ; always 0
  tail_header_start_block: u64           ; H + P
  canonical_metadata_hash: [u8; 32]
  footer_crc64: u64                      ; CRC-64/XZ over footer fields before footer_crc64
  zero padding to block_size
```

The footer is mandatory. If the primary header is unreadable, scan reconstruction reads the footer, derives `tail_header_start_block`, and validates the tail header/index copy.

### 2.5 Recovery behavior by sidecar metadata state

```text
primary valid, tail valid, hashes match:
  Sidecar usable.

primary valid, tail invalid/missing:
  Sidecar usable; emit SidecarMetadataCopyLost audit event.

tail valid, primary invalid/missing:
  Sidecar usable; emit SidecarPrimaryHeaderLost audit event.

footer valid, both header copies invalid:
  Sidecar structurally classifiable but not usable for parity recovery.
  Map reconstruction may continue; only this epoch is parity-unavailable.

footer invalid and primary invalid:
  Use bootstrap inline directory or parity_map directory if available.
  If directory can classify the tape file, map reconstruction may continue;
  parity recovery for this epoch is unavailable unless a header copy can be recovered.

all metadata sources unavailable:
  Catalog-less scan cannot classify this sidecar locally.
  If no directory entry covers it, map validation fails for the affected scope.
```

The important rule: **one damaged sidecar header must not poison catalog-less parity recovery for unrelated epochs.** At worst, the affected epoch becomes `SidecarMetadataUnavailable` and all other epochs remain recoverable.

### 2.6 Small final partial sidecars

For a small final partial epoch, `P` can be small enough that the primary copy, tail copy, and footer are physically close. This is accepted in v1:

```text
small final sidecar damage may destroy all local metadata copies;
bootstrap/parity_map directory still classifies the tape file structurally;
only that final partial epoch's parity becomes unavailable.
```

Do not add artificial padding solely to separate final-partial sidecar metadata. If the final epoch is small, the data at risk is small, and the three-copy archive policy remains the backstop.

## 3. Sidecar epoch directory and `parity_map` tape files

### 3.1 Problem

A whole-map digest can validate a reconstructed map, but cannot localize a single wrong entry. A damaged sidecar header can cause exactly one wrong entry and therefore fail the whole digest. The tape needs enough trusted structural information to correct sidecar classification during catalog-less scan.

v0.1 tried to put the full `SidecarEpochDirectory` in the bootstrap and deferred overflow handling. That is insufficient for conservative/custom schemes where the number of epochs can exceed a single bootstrap block's directory capacity.

### 3.2 Directory entry schema

The directory is a compact structural list of parity sidecars.

```cbor
SidecarEpochDirectory = {
    1: uint,                       ; directory_scope_tape_file_count
    2: uint,                       ; directory_scope_total_data_ordinals
    3: uint,                       ; directory_scope_highest_protected_ordinal
    4: bool,                       ; is_final_directory
    5: [* SidecarEpochDirectoryEntry],
}

SidecarEpochDirectoryEntry = {
    1: uint,                       ; tape_file_number
    2: uint,                       ; epoch_id
    3: uint,                       ; protected_ordinal_start
    4: uint,                       ; protected_ordinal_end_exclusive
    5: uint,                       ; sidecar_total_block_count
    6: uint,                       ; sidecar_header_block_count
    7: uint,                       ; parity_shard_block_count
    8: bytes .size 32,             ; canonical_metadata_hash
    9: uint,                       ; flags
}
```

Flags:

```text
0x01 = final partial epoch
0x02 = sidecar primary header was known good at write time
0x04 = sidecar tail header was known good at write time
```

The directory describes map structure, not media health. Do not include observed copy-health status in the canonical filemark-map digest.

### 3.3 Bootstrap payload extension

Bootstraps carry either an inline directory or a reference to a `parity_map` tape file.

```cbor
BootstrapPayload = {
    1: ParitySchemeRecord,
    ... existing fields ...,
   20: ?SidecarEpochDirectory,      ; inline, when it fits comfortably
   21: ?ParityMapReference,         ; present when directory is external
}

ParityMapReference = {
    1: uint,                        ; tape_file_number
    2: uint,                        ; block_count
    3: uint,                        ; directory_scope_tape_file_count
    4: uint,                        ; directory_scope_total_data_ordinals
    5: uint,                        ; directory_scope_highest_protected_ordinal
    6: bool,                        ; is_final_directory
    7: bytes .size 32,              ; parity_map_payload_sha256
    8: bytes .size 32,              ; canonical_map_digest for this scope
}
```

Rules:

```text
If the full directory fits within the bootstrap payload limit:
  MAY store it inline in field 20.

If the full directory does not fit comfortably:
  MUST write a parity_map tape file and store field 21.

A bootstrap without field 20 or 21 is usable for scheme discovery only.
It is not sufficient for robust catalog-less parity recovery.
```

### 3.4 Directory sizing rule

Do not hard-code assumptions such as "default tape has roughly 1100 epochs." The writer must compute the encoded directory size for the actual scheme and tape scope.

```text
inline_bootstrap_directory_limit =
    block_size
  - bootstrap_fixed_header_bytes
  - bootstrap_payload_crc_bytes
  - bootstrap_required_payload_bytes
  - conservative_padding_margin
```

Recommended margin: at least 4 KiB.

```text
if encoded(SidecarEpochDirectory).len <= inline_bootstrap_directory_limit:
    use inline directory
else:
    write parity_map tape file and reference it
```

This rule applies to default, conservative, and custom schemes. No production tape may omit the directory merely because it does not fit inline.

### 3.5 `parity_map` tape file kind

A `parity_map` tape file is a filemark-delimited control tape file outside `ParityDataOrdinal`, like bootstrap and sidecar tape files.

```text
parity_map tape file:
  block 0 .. M-1                 primary parity-map header + CBOR payload blocks
  block M .. 2M-1                tail parity-map copy
  block 2M                       parity-map footer locator
  filemark
```

For small maps, `M = 1`. For larger maps, `M = ceil(encoded_map_file_bytes / block_size)` plus any required header space.

The payload is canonical CBOR:

```cbor
ParityMapPayload = {
    1: tstr,                       ; format_id = "rem-parity-map-v1"
    2: bytes .size 16,             ; tape_uuid
    3: uint,                       ; sequence
    4: SidecarEpochDirectory,
    5: bytes .size 32,             ; canonical_map_digest for this scope
    6: ?tstr,                      ; writer_version
    7: ?tstr,                      ; write_timestamp
}
```

`parity_map` header/footer fields:

```text
magic = HMAC-SHA256(tape_uuid, b"REM\x00PMAP\x01")[0..8]
tape_uuid
sequence
payload_len
payload_sha256
canonical_map_digest
directory_scope_tape_file_count
directory_scope_total_data_ordinals
directory_scope_highest_protected_ordinal
is_final_directory
copy_kind = primary | tail
footer locator block with primary/tail starts and payload_sha256
CRC-64/XZ per header/footer block
```

### 3.6 Ordering and commit rule for `parity_map`

When a bootstrap needs an external map:

```text
1. Build the directory over the intended committed map scope.
2. Write parity_map tape file, including primary copy, tail copy, footer, and synchronous filemark.
3. Treat parity_map map entry as provisional until its filemark returns.
4. Encode bootstrap with ParityMapReference pointing to that parity_map.
5. Write bootstrap tape file and synchronous filemark.
6. Commit catalog_tape_files rows for parity_map + bootstrap in the same transaction.
```

The bootstrap's canonical filemark-map digest includes the structural tape-file entries for both the `parity_map` and the bootstrap itself, but does not include their content hashes or payload bytes. This avoids self-reference.

### 3.7 Reader behavior

On mount or catalog-less scan:

```text
if bootstrap has inline SidecarEpochDirectory:
    use it directly for directory overlay

else if bootstrap has ParityMapReference:
    locate/read the referenced parity_map if possible;
    verify payload_sha256 and canonical_map_digest;
    use its directory for overlay

else:
    bootstrap is scheme-only; catalog-less parity recovery is degraded
```

If the referenced `parity_map` is unreadable, the scanner may still reconstruct a map from sidecar primary/tail/footer metadata. The `parity_map` is a robustness accelerator and a damaged-sidecar-header repair source, not the sole path.

## 4. Catalog-less scan reconstruction algorithm

Replace the existing scan reconstruction with the following algorithm.

### 4.1 Inputs

```text
RawTapeSource
highest-sequence valid bootstrap payload
optional catalog_tape_files rows
```

If catalog rows are available and validate against tape identity, use the catalog as the primary filemark map. The scan path below is for catalog-less or catalog-suspect recovery.

### 4.2 Scan phase

For each physical tape file:

1. Count fixed blocks until filemark.
2. Store:
   ```text
   tape_file_number
   block_count
   first block bytes if readable
   last block bytes if readable
   any readable sidecar tail header/index blocks if footer identifies them
   any readable parity_map tail payload if footer identifies it
   ```
3. Classify in this order:
   ```text
   valid bootstrap magic + CRCs              -> bootstrap
   valid parity_map primary/tail + CRCs      -> parity_map
   valid sidecar primary header/index        -> parity_sidecar
   valid sidecar footer + tail header        -> parity_sidecar
   otherwise                                 -> object_candidate
   ```

Objects remain the residual class; do not require reading their pax header for classification, because the first object block may be the damaged block that parity is supposed to recover.

### 4.3 Directory overlay phase

After scanning, apply the best available directory source:

```text
1. final bootstrap inline directory, if present and valid
2. final bootstrap ParityMapReference, if referenced parity_map validates
3. highest-sequence intermediate inline directory or parity_map prefix
4. catalog rows, if operator has supplied a trusted catalog backup
```

For each directory entry:

```text
if scanned[tape_file_number].block_count == sidecar_total_block_count:
    force kind = parity_sidecar
    fill epoch_id and protected ordinal range from directory
    if no primary/tail header copy is valid:
        mark sidecar status = MetadataUnavailable
    else:
        require canonical_metadata_hash match
else:
    mark map as structurally inconsistent for this scope
```

This is the critical fix. A sidecar whose primary header is damaged is no longer demoted to object forever; the directory can recover its structural map entry.

### 4.4 Digest validation phase

Build the canonical filemark map projection after directory overlay. Then validate against the bootstrap's or parity_map's digest.

For a final bootstrap/map:

```text
scan_map_projection == final_map_digest
```

For an intermediate bootstrap/map:

```text
scan_map_projection_prefix(directory_scope_tape_file_count) == prefix_map_digest
```

If a tape file is identified by the directory but both sidecar header copies are unreadable, include it in the map projection as `kind=parity_sidecar` with the directory's structural fields. Do not include copy-health status in the canonical map digest; copy-health is recovery availability, not map structure.

### 4.5 Recovery eligibility

Map validity and sidecar usability are separate.

```text
Map-valid sidecar:
  The tape file is correctly known to be a sidecar with epoch/range/block_count.

Recovery-usable sidecar:
  At least one header/index copy is valid and the needed parity shard blocks pass CRC.
```

A map-valid but recovery-unusable sidecar causes only that epoch to be unavailable for parity recovery. It must not invalidate recovery for other epochs.

## 5. Sidecar emission, object commit bundles, and restart rebuild

### 5.1 Problem

Large objects can complete many epochs before the object filemark is written. If completed sidecars are held in volatile spool and deferred to later object boundaries, a crash can force restart to re-read many completed epochs to rebuild parity. v0.1's default `max_deferred_sidecars = 64` could make the rebuild window roughly 64 epochs, about 1 TiB at default geometry.

A simpler v1 rule is safer: do not defer completed sidecars across committed objects.

### 5.2 v1 policy: no volatile sidecar deferral

```text
Volatile sidecar deferral is prohibited in v1.
```

Completed epoch sidecars generated during an object must be emitted at that object's close, before the object becomes catalog-committed.

```text
At rem-tar object close:
  1. rem-tar writes tar EOF and BodyBlockWriter final post-EOF zero-fill.
  2. 3c writes the object tape-file's synchronous filemark.
  3. 3c emits every completed epoch sidecar generated by this object.
  4. 3c may write bracketing bootstraps during a long sidecar burst.
  5. Layer 5 commits the object + sidecars + bootstraps as one object commit bundle.
```

The only parity state allowed to remain unprotected after a committed object is the current **partial** epoch that has not yet reached a full sidecar.

```text
After any committed object bundle:
  total_committed_ordinals - highest_protected_ordinal < data_ordinals_per_epoch
```

This invariant is a hard v1 acceptance criterion.

### 5.3 Object commit bundle

Introduce the commit-bundle concept.

```rust
pub struct ObjectCommitBundle {
    pub object_tape_file: TapeFileEntry,
    pub sidecars_emitted: Vec<SidecarTapeFile>,
    pub parity_maps_written: Vec<ParityMapTapeFile>,
    pub bootstraps_written: Vec<BootstrapTapeFile>,
    pub highest_protected_ordinal_after: u64,
    pub total_committed_ordinals_after: u64,
    pub object_parity_state: ObjectParityState,
}
```

Rules:

```text
All tape files in the bundle are provisional until all their fixed blocks
and synchronous trailing filemarks have returned successfully.

Layer 5 commits the catalog rows for the whole bundle in one transaction.

There must be no catalog state where an object is committed but completed
sidecars generated by that object are absent merely because they were deferred.
```

If the process crashes during bundle emission, none of the bundle is catalog-committed. On restart, the append point is the trailing filemark of the previous catalog-committed tape file. The orphaned physical suffix is abandoned/truncated by the next append. The orchestrator retries the object from the beginning.

This trades rare crash-time rewrite cost for a bounded, simple restart invariant.

### 5.4 Sidecar burst policy

Sidecar bursts still occur for huge objects. v1 accepts this risk, because:

- sidecar metadata is now replicated and root-of-trust recoverable;
- losing some sidecar payloads degrades parity only for those epochs on that tape;
- the three-copy archive policy remains the protection against full sidecar-payload loss;
- avoiding bursts via volatile deferral creates worse restart behavior.

Policy:

```rust
pub struct SidecarBurstPolicy {
    /// Write a non-final bootstrap or parity_map+bootstrap after every N sidecars in a burst.
    /// Default: 8.
    pub bootstrap_every_sidecars: u32,

    /// If a single object is projected to produce more than this many sidecars,
    /// emit an audit warning and optionally require operator approval.
    /// Default: 64.
    pub large_object_sidecar_warning_threshold: u32,
}
```

There is no `max_deferred_sidecars` in v1. If future versions add deferral, they must satisfy §5.8.

### 5.5 Capacity and spool reserve

`begin_object(projected_size_blocks)` must reserve for:

```text
projected object blocks
+ object trailing filemark overhead
+ sidecars_completed_by_this_object_worst_case
+ header/tail/footer overhead for those sidecars
+ parity_map files needed for sidecar-directory overflow
+ bootstraps inserted during sidecar bursts
+ final partial epoch sidecar worst case
+ final parity_map/bootstrap if tape is closed after this object
+ safety margin
```

Spool reserve must cover completed sidecar payloads generated during the object until they are emitted at object close. For a very large object, this can be tens of GiB. If the sidecar spool reserve is insufficient, reject the object before writing.

### 5.6 Restart semantics

On restart:

```text
1. Load the committed catalog_tape_files prefix.
2. Find the highest committed tape_file_number.
3. Position to that tape file and verify tape UUID/kind/identity if readable.
4. Position after its trailing filemark.
5. Treat all later physical tape files as abandoned.
6. Rebuild FilemarkMapBuilder from the committed prefix.
7. Let W = highest_protected_ordinal.
8. Let T = total_committed_data_ordinals.
9. Assert T - W < data_ordinals_per_epoch for v1 committed state.
10. Re-read object blocks covering [W, T), rehydrate live partial-epoch accumulators.
11. Append new objects normally.
```

If `T - W >= data_ordinals_per_epoch` in a v1 catalog, treat it as catalog corruption or a legacy experimental tape. Do not silently perform a multi-epoch rebuild in production mode.

### 5.7 Crash windows

```text
Crash before object filemark:
  Object incomplete. No bundle commit. Restart appends after previous committed tape file.

Crash after object filemark but before all generated sidecars are written:
  Bundle incomplete. Object is not catalog-committed. Restart appends after previous committed tape file.

Crash after all sidecars written but before bundle DB commit:
  Bundle not catalog-committed. Restart appends after previous committed tape file.

Crash after bundle DB commit:
  Bundle committed. Restart append point is after the last committed tape file in the bundle.
  Rebuild only the trailing partial epoch if W < T.
```

This is stricter than committing every tape file as soon as its filemark returns. It is intentional: it keeps the committed prefix's rebuild cost bounded.

### 5.8 Future deferral rules, not v1

A future version may add sidecar deferral only if one of these is true:

```text
A. Deferred sidecar spool is crash-durable:
   sidecar payloads and metadata are fsynced locally;
   resume reloads them instead of rereading object data.

B. Multi-epoch rebuild is explicitly supported:
   catalog records a rebuild span;
   operator reserve/time budget is checked;
   tests prove bounded recovery time on real hardware.

C. Deferral is bounded to a very small value:
   max_deferred_sidecars <= 4 by default;
   production accepts the corresponding rebuild volume.
```

Until one of these is designed and tested, v1 must use no volatile deferral.

### 5.9 Large-object operational rule

If a single object would generate many sidecars, Layer 5 SHOULD split the source into multiple Remanence objects upstream, unless the operator explicitly accepts the cluster risk.

If rem-tar cannot split because the input is a single indivisible file, the write may proceed if it fits capacity and spool reserve, but the audit log must record:

```text
LargeObjectSidecarClusterRisk {
    object_id,
    projected_completed_epochs,
    projected_sidecar_cluster_bytes,
    bundle_commit_required: true,
}
```

## 6. Bulk / epoch recovery API

### 6.1 Problem

`recover_block_at()` is correct for isolated blocks, but inefficient for contiguous media damage. A contiguous damaged region can affect hundreds or thousands of stripes. Reconstructing block-by-block causes repeated LOCATEs and rereads of overlapping peers.

### 6.2 Required API

Keep the existing one-block API:

```rust
impl ObjectParitySource<'_> {
    pub fn recover_block_at(&mut self, body_lba: u64) -> Result<Vec<u8>, ParityError>;
}
```

Add region APIs:

```rust
pub struct RecoveryRegionRequest {
    pub tape_file_number: u32,
    pub body_lba_start: u64,
    pub block_count: u64,
    pub reason: RecoveryReason,
}

pub enum RecoveryReason {
    MediumError,
    LocateFailure,
    CrcMismatch,
    OperatorRequested,
}

pub struct RecoveredBlock {
    pub tape_file_number: u32,
    pub body_lba: u64,
    pub data: Vec<u8>,
    pub source: RecoverySource,
}

pub enum RecoverySource {
    CleanRead,
    Reconstructed,
    Unrecoverable,
}

impl ParitySource<'_> {
    pub fn recover_region(
        &mut self,
        request: RecoveryRegionRequest,
    ) -> Result<impl Iterator<Item = RecoveredBlock>, ParityError>;

    pub fn recover_ordinal_range(
        &mut self,
        ordinals: Range<ParityDataOrdinal>,
    ) -> Result<impl Iterator<Item = RecoveredOrdinalBlock>, ParityError>;
}
```

### 6.3 Planning algorithm

Bulk recovery must plan by epoch:

```text
1. Translate requested object BodyLba range to ParityDataOrdinal range.
2. Group missing/suspect ordinals by epoch_id and stripe_index.
3. For each affected epoch, load the sidecar header/index once.
4. Build the set of required peer reads for all affected stripes.
5. Deduplicate peer reads.
6. Sort peer reads by physical tape position to minimize seek/LOCATE churn.
7. Read clean peers in physical order.
8. Verify every peer against sidecar data CRCs or parity CRCs.
9. Treat failed CRCs as erasures.
10. Reconstruct all affected stripes in memory.
11. Verify every reconstructed data shard against sidecar data CRC.
12. Return recovered blocks in the caller-requested logical order.
```

### 6.4 Automatic escalation

A reader SHOULD automatically switch from `recover_block_at` to `recover_region` when any of these are true:

```text
>= 4 failed/suspect blocks within one epoch
>= 2 failed/suspect adjacent BodyLba blocks
operator requested a byte range overlapping a known damaged interval
read audit has already produced >= 4 RecoveryEvents on the same tape file
```

### 6.5 Memory-bounded epoch recovery

Bulk recovery can need a much larger memory footprint than the write path. The implementation must make this explicit and operator-bounded.

```rust
pub struct BulkRecoveryPolicy {
    /// Hard cap for in-memory recovery shards.
    /// Default: min(8 GiB, 25% of detected RAM), configurable.
    pub max_recovery_cache_bytes: u64,

    /// If true, planner may do multiple physical passes over an epoch
    /// rather than exceeding max_recovery_cache_bytes.
    pub allow_windowed_recovery: bool,

    /// Upper bound on affected stripes per recovery window.
    pub max_stripes_per_window: u32,
}
```

Planner rule:

```text
if all affected stripe peer sets fit under max_recovery_cache_bytes:
    recover the epoch in one physical-order pass
else if allow_windowed_recovery:
    split affected stripes into windows that fit the memory cap;
    recover each window separately;
    expect additional physical passes / seek cost
else:
    fail with RecoveryPlanExceedsMemoryBudget
```

This keeps the implementation safe on smaller hosts while allowing akash-class hosts to use larger recovery windows.

### 6.6 Cache shape

```rust
pub struct EpochRecoveryCache {
    pub epoch_id: u64,
    pub loaded_sidecar: SidecarHeaderIndexSet,
    pub data_shards: HashMap<(stripe_index, data_index), Vec<u8>>,
    pub parity_shards: HashMap<(stripe_index, parity_index), Vec<u8>>,
    pub max_bytes: u64,
}
```

This cache is for recovery mode only and should not be used on clean reads.

## 7. Drive compression precondition

### 7.1 Normative rule

For every 3c parity-protected write session:

```text
LTO hardware compression MUST be disabled.
Layer 3a MUST configure compression=false before any bootstrap/object/sidecar write.
Layer 3a MUST verify the effective drive compression mode after configuration.
If compression cannot be disabled or verified, the write session MUST fail before writing.
```

### 7.2 API addition

```rust
pub struct TapeWriteConfig {
    pub fixed_block_size: u32,
    pub drive_compression: bool,       // MUST be false for parity-protected writes
    pub hardware_encryption: EncryptionConfig,
}

pub enum ParityError {
    DriveCompressionEnabled,
    DriveCompressionModeUnknown,
    // ...
}
```

### 7.3 Bootstrap/catalog recording

Bootstrap and catalog must record:

```text
drive_compression = false
fixed_block_size = block_size
```

On mount, if a parity-protected tape records `drive_compression=true`, the reader must mark parity geometry as non-authoritative and refuse 3c recovery unless a future spec defines compressed-physical-extent semantics.

### 7.4 Hardware verification requirement

The exact SCSI page/field sequence must be proven on the deployed drive models.

Implementation requirement:

```text
Layer 3a must document the MODE SELECT / MODE SENSE page used to disable
and verify data compression on the actual LTO-9 drives.

The hardware proof must show:
  set compression false;
  read back effective compression state;
  external/manual re-enable is detected;
  parity write fails before BOT bootstrap if false cannot be verified.
```

Do not rely on a config variable alone. The drive's effective mode must be read back.

## 8. Reed-Solomon implementation policy

The implementation MUST own a reference GF(2^8) codec matching Appendix A:

```text
field polynomial: 0x11D
field element representation: byte value 0x00..0xff
Cauchy sets: Y_i = i, X_j = k + j
coefficient: inv(X_j xor Y_i)
parity accumulation: p_j += coeff(j,i) * data_i
```

`reed-solomon-erasure` may be used only if a conformance test proves byte-identical output and reconstruction against Appendix A vectors and randomized vectors. If the crate disagrees, the crate is disabled and the in-house codec is used.

Acceptance tests:

```text
1. Appendix A fixed vectors pass.
2. Randomized encode vectors pass against an independent slow GF implementation.
3. Incremental accumulation equals batch encode for random shard arrival orders.
4. Reconstruction passes for every erasure pattern of size <= m for small k/m.
5. Default k=128,m=4,S=512 smoke vectors pass.
```

## 9. Object size and no-spanning enforcement

rem-tar v1 does not support mid-object tape spanning. Layer 5 must enforce this before writing.

Add:

```rust
pub enum BeginObjectError {

        empty_tape_usable_blocks: u64,
        required_reserve_blocks: u64,
    },
    CapacityReserveExceeded { ... },
}
```

Before writing an object to any tape, Layer 5 must compute:

```text
projected_object_blocks
+ object_filemark_overhead

```

If the total cannot fit on an empty tape, the object is rejected and must be split upstream by the orchestrator.

## 10. Parity overhead terminology

Rename or document the two different overhead ratios:

```rust
impl ParityScheme {
    /// Parity bytes per usable data byte. For k=128,m=4: 3.125%.
    pub fn parity_over_data_ratio(&self) -> f64 { m as f64 / k as f64 }

    /// Parity bytes as a fraction of all data+parity bytes written.
    /// For k=128,m=4: 3.0303%.
    pub fn parity_fraction_of_total_written(&self) -> f64 {
        m as f64 / (k + m) as f64
    }
}
```

Operator-facing capacity planning should use `parity_over_data_ratio` when answering "how much extra tape do I need for a given amount of user data?" It should use `parity_fraction_of_total_written` when describing the composition of bytes already written.

## 13. Hardware and go-live proof gates

Mock tests and QuadStor tests are necessary but not sufficient. Before production live-tape use, run the following on scratch LTO media on the actual drive/library model.

### 13.1 Sidecar metadata damage tests

```text
1. Damage primary sidecar header block only:
   scan uses footer + tail header; map validates; epoch recovers.

2. Damage tail sidecar header only:
   scan uses primary header; map validates; epoch recovers.

3. Damage primary + tail header, footer intact:
   bootstrap/parity_map directory classifies sidecar; map validates;
   that epoch is SidecarMetadataUnavailable; unrelated epochs recover.

4. Damage primary + tail + footer:
   bootstrap/parity_map directory classifies sidecar if block_count matches;
   map validates; that epoch unavailable; unrelated epochs recover.

5. Remove/damage one sidecar tape file completely:
   map validates only if directory can account for the sidecar structurally;
   recovery for that epoch fails; unrelated epochs recover.
```

### 13.2 Catalog-less recovery tests

```text
1. Delete/ignore Postgres catalog.
2. Reconstruct filemark map by scanning tape.
3. Validate against final bootstrap digest or referenced parity_map digest.
4. Recover a damaged object block using sidecar information recovered from scan.
5. Repeat with one damaged sidecar header and prove unrelated epochs still recover.
```

### 13.3 Parity-map tests

```text
1. Force a scheme/directory size that cannot fit inline in bootstrap.
2. Writer emits parity_map tape file and bootstrap ParityMapReference.
3. Catalog-less scanner validates parity_map payload hash and map digest.
4. Damage inline bootstrap directory path; scanner still uses parity_map.
5. Damage parity_map primary copy; scanner uses tail copy.
6. Damage parity_map and one sidecar primary header; scanner falls back to sidecar tail/footer where possible.
```

### 13.4 Bulk recovery tests

```text
1. Inject a 64 MiB contiguous damaged range.
2. Inject a 512 MiB contiguous damaged range.
3. Inject a 513 MiB range and verify structured partial failure where applicable.
4. Compare per-block recovery time vs bulk recovery time.
5. Require bulk mode to avoid repeated reads of the same peer shard set.
6. Run with low max_recovery_cache_bytes and prove windowing works without exceeding memory cap.
```

### 13.5 Power-loss and append-resume tests

```text
1. Pull power before object filemark.
2. Pull power after object filemark but before generated sidecar burst completes.
3. Pull power mid-sidecar.
4. Pull power after all sidecars written but before bundle DB commit.
5. Pull power after bundle DB commit.
6. Restart and verify append point is the trailing filemark after the previous catalog-committed bundle when the current bundle was incomplete.
7. Restart and verify append point is after the current bundle when the bundle committed.
8. Prove committed-prefix rebuild volume satisfies:
      T - W < data_ordinals_per_epoch
```

### 13.6 Filemark durability test

A catalog-committed tape file is valid only if:

```text
all fixed blocks written
synchronous trailing filemark written with IMMED=0 or equivalent
post-barrier position captured
catalog row committed after the filemark returns
```

The implementation must prove on target hardware that a returned synchronous filemark is repositionable after power loss.

### 13.7 Drive compression test

```text
1. Attempt parity write with hardware compression enabled by external/manual configuration.
2. Layer 3a must detect and disable it before write.
3. If disabling or verifying fails, write must fail before BOT bootstrap.
4. Bootstrap and catalog must record drive_compression=false.
```

## 15. Acceptance criteria

Implementation is not production-ready until all of these pass:

```text
A. One damaged sidecar primary header does not cause FilemarkMapDigestMismatch.
B. One sidecar with both header copies damaged disables only that epoch's parity, not the whole tape's parity.
C. Catalog-less recovery works with the Postgres catalog absent.
D. Directory overflow uses parity_map; conservative/custom schemes do not lose sidecar-directory robustness.
E. No committed v1 catalog state can require restart rebuild of >= one full epoch.
F. Crash mid-object-bundle causes retry from previous committed bundle, not multi-epoch parity rebuild.
G. Bulk recovery of a 512 MiB contiguous region completes using each peer shard at most once per recovery window.
H. Bulk recovery respects max_recovery_cache_bytes.
I. Drive compression cannot be enabled for a parity-protected write session.
J. The implementation uses the in-house GF codec unless the external crate passes byte-identical conformance.
K. Large objects that cannot fit on an empty tape with reserves are rejected before writing any block.
L. Synchronous filemark durability is proven on the actual LTO drive model by power-loss test.
M. Open-epoch rebuild after restart is proven by power-loss test.
N. Final and intermediate bootstraps contain either inline sidecar epoch directories or references to validated parity_map tape files for their scope.
```

## 16. What remains accepted risk in v1

This addendum does not replicate the 512 MiB parity shard payload of every sidecar. If a contiguous damage event destroys the parity shard blocks for an epoch, that epoch's parity on this tape is lost and Layer 5 must fall back to another copy. That is acceptable because 3c is an improvement to each copy, not the sole archive redundancy mechanism.

This addendum also accepts sidecar clustering after huge objects. The v1 mitigation is:

```text
sidecar metadata replication;
sidecar directory / parity_map root-of-trust;
bracketing bootstraps during long bursts;
object commit bundles that keep restart rebuild bounded;
operator audit warnings for huge clusters;
three-copy archive fallback for sidecar-payload loss.
```

The alternative — volatile deferral — is explicitly not adopted in v1 because it worsens crash/restart behavior unless paired with durable parity spool or a much more complex rebuild budget.

This addendum removes the disproportionate failure where one damaged sidecar header disables catalog-less parity recovery for unrelated healthy sidecars, while keeping the implementation simple enough to start.
