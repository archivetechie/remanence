Good place to pause on the spec. The v0.1 is a fine anchor — detailed enough to commit to direction, vague enough to not lock in decisions you haven't earned the right to make yet.

Here's how I'd think about next steps, with some honest tradeoffs.

## Two valid directions, depending on what you want from this

**Direction A: More specs first.** Continue documenting before coding — manifest schema, OpenAPI, threat model, on-tape format details. Pro: forces clarity, produces artifacts that survive across collaborators, builds the political case (Remanence already *looks* like a real project). Con: easy to over-specify before you've touched the hardware, and detailed specs written from theory often need rewriting once code meets reality.

**Direction B: Start building.** Set up the Rust workspace, get QuadStor running on akash, write the first SCSI commands, capture real responses. Pro: every assumption gets validated against reality fast, momentum is motivating, you'll find spec gaps that you couldn't have anticipated. Con: less to show non-technical readers, harder to share with collaborators until something is demonstrable.

My honest recommendation: **start building, but lightly**. You're an experienced sysadmin doing this on the side — momentum matters more than polish at this stage, and code will tell you things the spec can't. Specifically:

## Concrete next steps, in order

### Step 1: Set up QuadStor on akash (half a day)

Install QuadStor VTL on akash, configure it to expose a virtual MSL-class library with 4 virtual LTO-9 drives and a 40-slot virtual changer. Verify `lsscsi -g` shows the expected device nodes. Run `sg_inq` and `mtx -f /dev/sgN status` against the virtual devices to confirm the virtual library behaves like real hardware.

This is the foundation for everything else. Without it, you'd be developing blind against fixtures. With it, you have a real-feeling tape library you can break and reset freely.

### Step 2: Capture real-hardware fixtures from the MSL3040 (one window of access)

The next time you have any access to the production MSL3040 — even briefly, even just for read-only operations — capture as many real SCSI responses as you can. Specifically:

```bash
sg_inq -v /dev/sgN          # for the changer and each drive
sg_inq -p 0x80 /dev/sgN     # VPD page 0x80 (unit serial)
sg_inq -p 0x83 /dev/sgN     # VPD page 0x83 (device identifiers)
sg_raw -v -r 64k /dev/sg_changer b8 10 00 00 ff ff 00 ff ff 00 00 00   # READ ELEMENT STATUS with DVCID
sg_logs -a /dev/sg_drive    # all log pages
sg_logs --page=0x17 /dev/sg_drive  # volume statistics (if a tape is loaded)
sg_logs --page=0x2E /dev/sg_drive  # tape alert
```

Save the raw binary responses to files. These become your test fixtures — every parser you write later can be tested against real production hardware behavior, even when you're working from akash with no library attached. This is the highest-leverage activity in the whole project right now, because access windows may be irregular.

If you can't get even read-only access to the MSL3040 right now, QuadStor fixtures will get you 80% of the way; the remaining 20% is the vendor-specific quirks that only real hardware exposes.

### Step 3: Rust workspace skeleton (one evening)

Set up the multi-crate workspace on akash:

```
~/remanence/
├── Cargo.toml              # workspace root
├── crates/
│   ├── remanence-scsi/     # Layer 1: SCSI core, no_std-friendly
│   ├── remanence-library/  # Layer 2: runtime topology, udev
│   ├── remanence-tape/     # Layer 3: tape operations, on-tape format
│   ├── remanence-catalog/  # Layer 4: postgres, sqlx
│   ├── remanence-api/      # Layer 5: axum, mTLS, SSE
│   ├── remanence-cli/      # the `rem` binary
│   └── remanenced/         # the daemon binary
├── fixtures/
│   ├── inquiry/
│   ├── element-status/
│   └── log-sense/
├── docs/
│   └── spec-v0.1.docx      # check in the spec
├── README.md
└── LICENSE                 # AGPL-3.0 is the safe default for archive infrastructure
```

A minimal `Cargo.toml` with `tokio`, `serde`, `serde_json`, `bytes`, `thiserror`, `tracing`, `nix`. Don't worry about being complete; you'll add as you go.

Create a git repo, push it somewhere private (or public if you're ready). Sourcehut, Codeberg, or a private GitHub repo are all fine for this stage.

### Step 4: First SCSI command — INQUIRY (one evening)

The smallest meaningful piece of real code: send a standard INQUIRY (CDB `0x12`) to a `/dev/sgN`, parse the response, return a struct with the vendor, product, revision, and device type. Test against QuadStor's virtual drives.

This sounds trivial, and it is — but it forces you to confront the entire toolchain in miniature: `SG_IO` ioctl semantics, sense data handling, error types, the testing pattern with fixtures. Get it clean and the rest of Layer 1 follows the same pattern with more bytes.

Maybe a hundred lines of Rust including tests. One evening.

### Step 5: VPD page 0x80 (one evening)

INQUIRY with EVPD bit set and page code 0x80, parse the serial number. Now you can identify drives by their actual serial — the foundation for the join-by-serial logic in Layer 2.

### Step 6: READ ELEMENT STATUS (a weekend)

This is the meaty one and the most rewarding. CDB `0xB8`, DVCID bit set, parse the element descriptors. After this, your code can enumerate the entire library state and identify which drive is in which bay by serial number. Test against QuadStor's virtual changer.

At the end of this step, you have something real and demonstrable: a Rust program that, run on akash, prints a live topology view of the QuadStor virtual library. That's a meaningful milestone — the first thing worth showing anyone.

### Step 7: Stop and reassess

Two to four evenings of work, plus one access window with the MSL3040, gets you to: working Layer 1 fundamentals tested against virtual hardware and real-hardware fixtures, a clean workspace, and the foundation of the join-by-serial logic.

At that point, take stock. Is the design holding up? Are the SCSI abstractions clean? Is anything in the spec wrong now that you've seen real responses? Update spec v0.2 with corrections. *Then* decide whether to push further on code (move into Layer 2, libudev events) or back to spec (manifest format, OpenAPI), depending on what feels most uncertain.

## What I'd explicitly defer right now

A few things that are tempting to work on early but pay off later:

- **The Postgres schema.** You'll redesign it three times once you've seen real data flowing through Layer 1-3. Don't write the SQL until you have something to put in it.
- **The API.** Same reason. The right resource model emerges from working operations, not from sketching.
- **The threat model as a separate doc.** Worth writing eventually, but the security section in the v0.1 spec is enough for now. Revisit when you have running code worth threatening.
- **Logo, branding, project website.** All of this is fun procrastination. Defer until there's running code worth pointing at.
- **The Python client library.** Doesn't make sense to design until the API exists.
